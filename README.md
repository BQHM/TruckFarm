# TruckFarm 农场经营数字沙盘

TruckFarm 是一个基于毕业设计旧项目持续迭代升级的农场经营数字沙盘。旧版项目始于 2023 年 10 月，完成于 2024 年 1 月，最初使用 Servlet、MiniUI、JDBC 和 MySQL 实现了登录、用户管理、部门管理、农作物管理等基础后台功能。

这次升级不是完全另起炉灶做一个新项目，而是在旧项目基础上边重构、边检查旧代码不规范之处、边学习新技术替换旧技术：用 Java 25、Spring Boot 4、PostgreSQL、Flyway、Spring Security、Vue 3 等现代技术逐步替代 Servlet、JDBC、Session、MiniUI、手工 SQL 等旧实现，同时把项目从普通管理平台升级为带时间推进、天气环境、作物生长、随机事件、资金变化和经营复盘的模拟系统。

> 当前项目处于迭代升级规划和技术验证前阶段。旧版代码会继续保留为学习、对照和迁移源；新版采用“管理系统是底座，模拟引擎是核心亮点，经营决策是主线”的设计思路。

> 需求与任务已经按 Codex skills 校准：docs/requirements.md 作为 spec-driven-development 的需求源，docs/tasks.md 作为 planning-and-task-breakdown 的阶段计划和开发留痕。

## 项目定位

TruckFarm 面向学习、演示和简历展示场景，模拟一个中小型农场从种植、采购、库存、销售到经营复盘的全过程。它不是纯游戏，也不是通用后台模板，而是一个具备真实业务数据、时间推进、环境变化和非固定结果机制的农场经营仿真系统。

核心目标：

- 管理底座：作物、地块、供应商、客户、库存、采购、销售、用户权限、资金流水可维护。
- 环境模拟：日期、季节、天气、温度、湿度、市场行情会随时间变化。
- 非固定结果：采购、播种、养护、收获、销售等操作受环境、规则、随机种子和 AI 修正影响。
- 经营闭环：播种 -> 生长 -> 事件 -> 收获 -> 入库 -> 销售 -> 资金变化 -> 经营复盘。
- 可解释模拟：每次模拟结果都记录上下文、基础规则、AI 修正、随机种子和最终结果。
- AI 定位：AI 只做环境影响分析、修正因子建议和结果解释，不直接决定核心业务数据。

一句话概括：

```text
不是只管理农场数据，而是模拟农场经营过程。
```

## 核心模拟机制

用户操作不会直接产生固定结果，而是经过以下流程：

```text
用户操作
  -> 收集当前上下文
  -> 本地规则计算基础范围
  -> AI 生成环境修正因子和解释
  -> 后端校验和裁剪 AI 输出
  -> 使用 random_seed 随机采样
  -> 生成最终结果
  -> 写入业务表、模拟结果表、资金流水和事件日志
```

示例：采购 100kg 番茄种子时，最终优品率不固定，而是受供应商信誉、当前天气、运输距离、季节、市场紧缺程度影响。

```text
基础规则：优品率 78% - 94%
AI 修正：连续阴雨导致受潮风险上升，优品率 -4%，损耗率 +2%
随机采样：优品率 84%，普通品率 12%，损耗率 4%
最终结果：优品 84kg，普通 12kg，损耗 4kg，生成运输湿度偏高事件
```

AI 失败时系统必须降级为本地规则 + 默认修正 + 随机采样，不能阻塞采购、播种、收获、销售等核心流程。

## 计划中的项目亮点

后续实现完成后，TruckFarm 将重点展示以下能力：

- 从传统 Servlet 项目重构到 Java 25 + Spring Boot 4 + Vue 3 前后端分离架构。
- 使用 PostgreSQL + Flyway 管理新版业务表、模拟表、事件表、资金流水表和经营复盘表。
- 设计 Simulation Engine，通过时间 Tick 推动作物生长、天气变化、地块状态、市场行情和经营状态变化。
- 设计“规则引擎 + AI 修正 + 随机种子”的非固定结果机制，使采购、播种、收获、销售受环境影响。
- 将 AI 作为环境影响分析器，只返回结构化修正因子和解释，最终结果由后端校验、采样和落库。
- 通过 `simulation_outcome`、`farm_event`、`random_seed` 和 `fund_flow` 记录完整模拟过程，实现结果可追踪、可复现和可降级。
- 通过事务保证收获入库、销售出库、库存流水、资金流水和事件日志的一致性。
- 使用 Vue 3 + ECharts + SVG / Canvas 构建首页沙盘、地块沙盘、事件时间线和经营复盘看板。
