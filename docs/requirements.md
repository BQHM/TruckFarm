# TruckFarm 数字沙盘需求说明

版本：v0.3
日期：2026-05-13
状态：按 spec-driven-development 校准后的需求规格草案

## 0. 规格总览（按 spec-driven-development）

本文件是 TruckFarm 当前需求源，按 `spec-driven-development` 的规格模板维护。进入实现前，需求变更先改本文件，再同步 `docs/tasks.md` 的任务和验收条件。

### 0.1 Objective

建设一个基于旧 Servlet + MiniUI 项目迭代升级的农场经营数字沙盘。成功状态不是“多做几个后台 CRUD”，而是能演示一条可追踪、可复现、可解释的农场经营闭环：采购种子、播种、推进日期、触发环境事件、收获入库、销售出库和经营复盘。

### 0.2 Tech Stack

| 层级 | 目标技术 | 当前约束 |
| --- | --- | --- |
| 后端 | JDK 25、Spring Boot 4、Spring Security、JWT、MyBatis 3、MapStruct、Flyway | P1 先做兼容性验证，未通过的依赖不得直接写入核心业务实现 |
| 数据 | PostgreSQL、Redis 7 | PostgreSQL 是事实数据源，Redis 只做缓存、黑名单和临时状态 |
| 前端 | Vue 3、TypeScript、Vite、Element Plus、Pinia、Vue Router、Axios、ECharts | 页面不得退化为默认后台模板，首页必须体现沙盘叙事 |
| 工程化 | Docker Compose、JUnit 5、Mockito、AssertJ、GitHub Actions | 先保证本地可运行，再补 CI 和部署包装 |
| AI | 统一 `AiClient` / Mock Provider / 结构化 JSON 输出 | AI 只给修正因子和解释，不能直接写核心业务事实 |

### 0.5 Code Style

后端请求对象优先使用 `record`，Controller 只做参数校验和调用 Service，返回 `Result<T>`，不直接返回 Entity。

```java
public record CreateCropRequest(
  @NotBlank String name,
  @NotNull Long categoryId,
  @Positive Integer growthCycleDays,
  @NotNull @DecimalMin("0.00") BigDecimal referencePrice
) {}
```

前端页面使用 `<script setup lang="ts">`，接口集中放在 `src/api/`，页面不直接拼接 URL，不大量使用 `any`。

### 0.7 Boundaries

- Always：先更新需求和任务再实现；核心金额、数量、面积使用 `BigDecimal` / PostgreSQL `NUMERIC`；核心业务保留审计和模拟留痕；AI 失败必须降级。
- Ask first：移动或删除旧版 `src/`、`pom.xml`、`table.sql`；新增大型依赖；改变阶段优先级；引入 MyBatis Plus、SpringDoc 等兼容性未验证能力。
- Never：在旧 Servlet 架构继续堆新功能；明文密码；Session 作为新版正式鉴权；AI 直接修改库存、资金、订单状态；物理删除核心历史业务数据；提交真实密钥。

### 0.8 Success Criteria

- P1 成功：Java 25 + Spring Boot 4 最小后端、PostgreSQL、Flyway、Redis、前端 Vite 项目和 Docker Compose 均可本地启动，验证结论写入 `docs/tasks.md`。
- MVP 成功：可以按固定演示路径完成登录、采购、播种、推进日期、事件触发、收获入库、销售出库和经营复盘。
- 沙盘成功：动作结果页能展示输入上下文、基础规则、AI 修正或 fallback、最终结果和 `random_seed`。
- 工程成功：核心 Service 有测试，Docker Compose 可启动基础环境，README 有启动步骤、演示账号和截图。
- 迁移成功：旧用户、部门、作物、菜单等概念有明确映射，旧问题不会原样带入新版。

### 0.9 Open Questions

- 前端沙盘视图首版使用地块卡片、SVG 平面图还是 Canvas，需要在 P7 前结合实现成本确认。

## 1. 项目背景

TruckFarm 原始版本是基于 Servlet、MiniUI 和 MySQL 的毕业设计项目，功能集中在登录、用户、部门、农作物管理等基础后台能力。本次不是纯粹新开发，而是在旧项目基础上进行迭代升级：边检查旧代码不规范之处，边学习 Java 25、Spring Boot 4、PostgreSQL、Flyway、Spring Security、Vue 3 等新技术，边将旧功能重构为现代前后端分离架构，并补足完整业务闭环、模拟机制和工程化能力。MyBatis / MyBatis Plus、OpenAPI 等生态依赖需要在 P1 阶段先做 Spring Boot 4 兼容性验证。

## 2. 产品定位

TruckFarm 面向学习、演示和简历展示场景，模拟中小型农场从种植、采购、库存、销售到经营复盘的全过程。系统保留作物、地块、库存、采购、销售等管理底座，但核心方向升级为带时间推进、天气环境、事件规则、资金变化和非固定操作结果的农场经营数字沙盘。

目标：

- 作物资料可维护，并扩展生长周期、适宜季节、环境敏感度等模拟参数。
- 地块状态可追踪，并包含土壤湿度、肥力、病虫害风险等环境状态。
- 模拟日期可推进，天气、市场价格、作物生长、库存质量和资金状态会随时间变化。
- 采购、播种、养护、收获、销售等操作产生非固定结果。
- AI 根据当前环境输出修正因子和结果解释，但不直接决定核心业务数据。
- 首页沙盘能展示当前农场状态、地块沙盘、事件时间线、经营结果和经营资金。
- 权限、日志、接口文档、Docker 部署具备简历项目完整度。

## 4. MVP 范围

第一版优先打通以下闭环：

```text
登录 -> 初始化农场 -> 查看天气和地块 -> 采购种子 -> 播种 -> 推进日期 -> 触发事件 -> 收获入库 -> 销售 -> 经营复盘
```

MVP 模块：

- 认证授权：登录、JWT、刷新 Token、退出登录、密码加密。
- 用户权限：用户、角色、菜单、角色菜单授权、动态路由。
- 组织员工：部门、员工资料、员工状态。
- 作物管理：作物分类、作物档案、生长周期、参考价格、适宜季节、环境敏感度、生长规则。
- 地块管理：地块档案、面积、位置、状态、土壤湿度、肥力、病虫害风险。
- 种植计划：计划创建、开始种植、完成收获、取消计划。
- 采购管理：供应商、采购单、采购明细、采购结果模拟、采购入库。
- 库存管理：当前库存、出入库流水、库存预警、盘点调整。
- 销售管理：客户、销售订单、销售明细、销售出库。
- 资金管理：经营账户、收支流水、采购支出、销售收入、手工调整。
- 数据看板：经营概览、地块沙盘、天气环境、事件时间线、销售趋势、资金趋势和风险预警。
- 模拟引擎：模拟日期、天气生成、时间 Tick、规则计算、随机采样、结果记录。
- 事件日志：天气事件、病虫害、市场波动、库存损耗、操作影响。
- 系统日志：登录日志、操作日志。

二期模块：文件附件、经营复盘增强、通知提醒、多农场能力。AI 辅助模拟可以在本地规则闭环稳定后接入。

## 5. 核心功能

### 5.1 认证授权

- 用户使用账号密码登录。
- 密码使用 BCrypt 等安全算法加密存储。
- 登录成功返回 Access Token 和 Refresh Token。
- 退出登录后 Token 加入 Redis 黑名单。
- 后端返回当前用户、角色、权限和菜单。
- 未登录返回 401，无权限返回 403。

### 5.2 用户角色菜单

- 用户可绑定多个角色。
- 角色可绑定菜单和按钮权限。
- 前端根据后端菜单生成侧边栏和路由。
- 用户、角色、菜单变更需要记录操作日志。

### 5.3 作物与地块

- 作物支持分类、名称、生长周期、参考价格、单位、状态。
- 作物需要补充生长规则、适宜季节和环境敏感度。
- 地块支持编号、名称、面积、位置、土壤类型、状态。
- 地块状态包括空闲、种植中、休耕、停用。
- 被业务引用的作物和地块禁止物理删除。

### 5.4 种植计划

- 计划必须绑定作物、地块和负责人。
- 状态流转：待开始 -> 种植中 -> 已收获，可取消。
- 预计收获日期不得早于计划开始日期。
- 同一地块同一时间段不能存在冲突的进行中计划。
- 完成收获时生成收获记录，保留产量、损耗和收获日期。

### 5.5 采购库存

- 采购单绑定供应商和采购负责人。
- 采购单包含多个明细，入库后增加库存并生成库存流水。
- 库存数量不得为负。
- 支持库存预警阈值，低于阈值展示在首页沙盘。

### 5.6 销售订单

- 销售单绑定客户和销售负责人。
- 销售单包含多个明细，出库时扣减库存并生成库存流水。
- 库存不足时禁止出库。
- 已取消订单不参与销售收入统计。

### 5.7 资金与经营收支

- 系统维护农场经营账户和当前可用余额。
- 采购入库默认产生资金支出，销售出库默认产生资金收入。
- 手工充值、赔付或调整需要保留资金流水。
- 首页沙盘和经营复盘按资金流水统计收入、成本和现金变化。

### 5.8 首页沙盘

首页展示：当前模拟日期、季节、天气、资金、地块状态、作物生长进度、风险事件、库存预警、市场价格、收益趋势和待处理事项。

### 5.9 模拟引擎

- 系统维护当前模拟日期、季节和经营轮次。
- 支持手动推进一天，后续可扩展自动推进或推进一周。
- 每次 Tick 会生成或读取天气记录，并更新地块状态、作物生长状态、库存质量、市场价格和经营环境。
- 每次 Tick 必须记录模拟日志，方便追踪和复盘。
- 同一个操作在不同环境下允许产生不同结果，但必须保存随机种子和结果上下文。

### 5.10 非固定操作结果

- 采购结果包括优品率、普通品率、损耗率、延迟天数和风险事件。
- 播种结果包括出芽率、初始生长速度、种子损耗和病害风险。
- 养护结果包括水分恢复、肥力变化、生长速度修正和过量风险。
- 收获结果包括实际产量、优品率、损耗率和入库质量等级。
- 销售结果包括成交价格、成交数量、客户满意度和退货风险。
- 所有结果必须由本地规则给基础区间，AI 只能提供修正因子和解释。

### 5.11 AI 辅助模拟

- AI 输入包括操作类型、天气、季节、地块状态、供应商或客户参数、规则基础区间。
- AI 输出必须是结构化 JSON，例如优品率修正、损耗率修正、风险等级、事件标题和原因解释。
- 后端必须校验 AI 输出范围，超出范围时裁剪或 fallback。
- AI 调用失败时使用本地规则 + 默认修正，不影响核心流程。
- AI 调用上下文、输出、状态和错误信息需要保存，便于审计和调试。

### 5.12 经营复盘

- 每轮经营结束后生成收入、成本、净利润、库存损耗、资金变化、风险事件和关键决策影响。
- 复盘报告需要说明哪些天气、事件和用户操作影响了产量或收益。
- AI 可以基于复盘数据生成自然语言总结，但不能修改复盘事实数据。

## 6. 非功能需求

| 类别 | 要求 |

| 可测试性 | 核心 Service、模拟引擎、规则计算、随机种子复现和关键接口有测试 |
| 可展示性 | README 有截图、演示账号、启动步骤和项目亮点 |

## 7. 不做范围

- 不做真实支付、物流和电商交易。
- 不接入真实农业传感器和物联网设备。
- 不做复杂财务总账和会计凭证。
- MVP 不做多租户 SaaS。
- AI 功能不作为核心业务强依赖。
- 不让 AI 直接修改库存、资金、订单状态等核心数据。
- 第一版不做复杂 3D 地图和纯游戏化操作。

## 8. 验收标准

- 可以登录系统并看到动态菜单。
- 可以完成采购、播种、推进日期、触发事件、收获入库、销售和复盘的完整演示流程。
- 首页沙盘能基于演示数据展示当前日期、天气、地块状态、事件时间线、库存、资金和收益。
- OpenAPI 文档可访问。
- Docker Compose 可以启动基础环境。
- README 有启动步骤、演示账号和截图。
